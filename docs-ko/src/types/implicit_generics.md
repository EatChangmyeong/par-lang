# 암시적 제네릭

Par에서는 이미 [전칭](./forall.md)으로 명시적 제네릭을 지원하고 있다.

```par
module Main

import {
  @core/Int
  @core/String
}

dec Swap : [type a, type b, (a, b)!] (b, a)!
def Swap = [type a, type b, pair]
  let (first, second)! = pair
  in (second, first)!
```

`forall`을 사용하면 호출자가 `Swap`을 사용할 때 명시적으로 타입을 전달해야 한다.

```par
def Pair = ("Hello!", 42)!
def Swapped = Swap(type String, type Int, Pair)
```

완전히 명시적인 스타일로 엄밀하게 작성했다. 하지만 대부분의 일상적인 상황에서는 전달하는 값만 보고 타입을 추론할 수 있다. 이런 경우를 위해 **암시적 제네릭** 역시 지원하고 있다.

## 문법

암시적 제네릭이라는 타입이 별도로 있는 것은 아니다. `<a, ...>` 대입자는 **[함수](./function.md) 타입**의 확장으로, 해당하는 타입의 `[...]` 직전에 작성해야 한다.

```par
<a, b>[Arg] Res
```

분기, 선택, 박스 등 다른 타입의 앞에는 `<a>`를 붙일 수 없다.

암시적 제네릭의 핵심 규칙은 국소적 추론이다.

**암시적 타입 변수는 `<...>` 대입자 바로 뒤에 오는 인자 하나만으로 추론할 수 있어야 한다.**

위의 규칙을 문법에서도 확인할 수 있다. `<...>` 다음에 오는 대괄호에는 인자 타입이 하나만 올 수 있다. 값 인자가 여러 개 필요할 경우에는 커링된 함수 타입을 작성하면 된다.

즉, 아래의 코드는 문제가 없지만...

```par
dec Concat : <a>[List<a>] [List<a>] List<a>
```

아래의 코드는 문법 오류가 된다.

```par
dec Concat : <a>[List<a>, List<a>] List<a>
```

단, 호출할 때는 모든 인자를 한 번에 전달해도 무관하다.

```par
Concat(left, right)
```

타입 추론은 첫 인자에만 의존하기 때문에, 호출 방식에 상관없이 타입은 커링된 형태로 작성해야 한다.

한편, 암시적 타입 인자를 '수동으로 지정'하는 문법 역시 지원하지 않는다. API를 설계할 때는 각각의 타입 변수마다 암시적 추론과 명시적 추론 중 하나를 선택해야 한다.

- 암시적(`<a>`) 제네릭은 무조건 추론된다.
- 명시적(`[type a]`) 제네릭은 무조건 호출자가 명시해야 한다.

이는 함수 호출을 일관적으로 만들기 위한 의도적인 설계이다.

암시적 타입 매개변수에는 `<a: box>`나 `<a: data>`와 같이 제약을 추가할 수도 있다. 타입 제약에 관해서는 [타입 제약](./constraints.md)에서 다룬다.

## 생성

앞에서 다룬 `Swap` 예제를 암시적으로 만들어 보자.

```par
dec Swap : <a, b>[(a, b)!] (b, a)!
```

이 타입의 값을 생성할 때는 값 단계에서도 암시적 타입 변수를 작성한 뒤에 값 인자를 받아야 한다.

```par
dec Swap : <a, b>[(a, b)!] (b, a)!
def Swap = <a, b>[pair]
  let (first, second)! = pair
  in (second, first)!
```

전칭 타입과 문법이 다른 것은 의도된 것이다. 암시적 제네릭과 명시적 전칭 제네릭은 서브타입 관계가 없는 서로 다른 타입이다.

타입 변수를 복잡한 인자 안의 깊숙한 곳에서 사용하고 있을 때도 암시적 제네릭을 사용할 수 있다.

```par
dec Flatten : <a>[List<List<a>>] List<a>
```

## 소멸

암시적 제네릭을 사용할 때는 일반적인 함수처럼 호출하면 된다.

```par
def Swapped = Swap(Pair)
```

이때 `<a, b>` 바로 다음에 오는 인자인 `Pair`에서 `a = String`과 `b = Int`가 추론된다.

타입 추론을 직접 조작해야 한다면 아래와 같이 원하는 식에 타입을 매기는 등의 방법으로 국소적으로 타입 표기를 추가하면 된다.

```par
type T in expr
```

예를 들어 보자. (다소 부자연스럽지만 이렇게 사용한다는 것만 알고 넘어가자.)

```par
Concat(type List<Nat> in *(), numbers)
```

`Concat(*(), numbers)`라고만 작성하면 빈 리스트 `*()`에 의해 `a`가 불가능한 타입인 `either {}`로 추론되므로, 원하는 타입인 `Nat`이 되지 않는다.

## 고차 인자

암시적 제네릭은 *인자 값*으로부터 타입 변수를 추론한다. 이 방식은 원래 인자 전체의 타입이 결정되어 있을 때 가장 잘 작동하고, 추가로 타입을 부분적으로만 알 수 있는 식을 검사할 때는 인자로 와야 할 타입의 정보를 활용하기도 한다.

익명 함수가 대표적인 예시에 해당한다. 이때 `box`는 `Map`이 박싱된 함수를 받기 때문에 추가한 것으로, 여기서는 중요하지 않다.

```par
dec Map : <a>[List<a>] <b>[box [a] b] List<b>
```

리스트 인자에서 `a`가 추론된다. 매퍼 함수를 검사할 때는 `b`를 아직 알 수 없지만, 검사기에서는 부분적으로 예측한 타입 `box [a] b`를 활용할 수 있다. 즉, 람다 매개변수에 타입 `a`가 부여된 채로 람다 본문의 타입 방정식을 풀어서 `b`를 추론한다.

```par
Map(numbers, box [x] `#{x}`)
```

여기서 `x`의 타입은 `numbers`의 원소 타입으로 검사할 수 있고, 문자열 보간에서 `b = String`이 추론된다.

단, 인자만으로 타입을 추론하기 어려울 경우에는 국소적 타입 표기를 작성해야 하기도 한다. 보통 빈 자료구조가 이에 해당한다.

```par
Concat(type List<Nat> in *(), numbers)
```

암시적 타입 인자를 수동으로 전달하는 문법은 없기 때문에, 이 경우에도 타입 표기는 암시적 타입 매개변수가 아니라 전달되는 값에 작성했음을 알 수 있다.

## 암시적 제네릭과 순서쌍

[순서쌍](./pair.md) 타입에도 암시적 제네릭이 있으며, [존재](./exists.md) 타입의 암시적 버전에 해당한다.

함수와 비슷하게 `<a, ...>` 대입자가 따라붙는 순서쌍 타입에 종속되며, 해당하는 순서쌍 타입의 `(...)` 직전에 작성해야 한다.

단순한 예제를 확인해 보자. `AnyDrop`은 어떤 알 수 없는 타입의 값 하나를 보관하면서, 추가로 그 값을 버리는 작은 '가상 함수 테이블' (박싱된 [선택](./choice.md) 값)을 가지고 있다.

```par
type AnyDrop = <a>(a) box choice {
  .drop(a) => !,
}
```

이 타입은 존재 타입의 암시적 버전에 해당한다.

```par
type DropMe = (type a, a) box choice {
  .drop(a) => !,
}
```

### 생성

`AnyDrop`을 생성할 때 숨은 타입을 작성하지 않는 것을 확인할 수 있다.

```par
def Example: AnyDrop = (7) box case {
  .drop(_) => !,
}
```

### 소멸

쌍대의 관점에 의해, 암시적 제네릭 순서쌍을 풀어낼 때는 지역 타입 변수를 작성해야 한다.

```par
let <a>(value) vtable = Example
vtable.drop(value)
```

이는 암시적 제네릭 함수의 '거울상'으로 생각할 수 있다. 함수의 경우에는 생성 시에 `<a, ...>`를 작성하고 호출 시에 추론이 일어나지만, 여기서는 별도의 `a` 없이 생성하는 대신 소멸 시에 `a`를 작성한다.
