# 정의와 선언

`par new`로 생성한 패키지에서 계속 작업해 보자.

`src/Main.par` 파일의 첫 줄은 다음과 같다.

```par
module Main
```

이외에도 코드의 맨 위에서 다른 모듈을 가져올 수 있지만, 일단은 무시하고 그 다음에 오는 내용만을 살펴 보자.

Par 모듈의 최상위에는 다음과 같은 내용을 작성한다.

- **정의**
- **선언**
- **타입 정의**

이들은 모듈 전체에서 무한정 사용할 수 있는 **전역 이름**의 정의가 된다.

새로운 패키지에서 `par run`을 하면 `Main` 모듈의 `Main` 정의, 즉 `Main.Main`을 찾는다.

즉,

- `par run`을 하면 `Main.Main`이 실행된다.
- `par run Main.Other`를 하면 `Main` 모듈의 `Other` 정의가 실행된다.

모듈이 늘어나면 `par run util/Parse.Program`과 같은 경로 역시 사용할 수 있다. 이에 대해서는 다음 장에서 알아보도록 하자.

> Par는 선형 타입 시스템을 가지므로, 어떤 지역 변수에는 *반드시 한 번* 사용해야 한다는 제한이 있다. 선형 타입을 가지는 변수가 이에 해당한다. 전역 정의는 타입에 상관 없이 임의의 횟수만큼 사용할 수 있다.

Par의 작명 규칙은 단순하다.
- **전역 이름은 대문자로 시작한다.** 전역 타입이나 함수 등이 이에 해당한다.
- **지역 이름은 소문자나 `_`로 시작한다.** 지역 변수나 함수 매개변수, 제네릭 함수의 타입 변수 등이 이에 해당한다.

전역 이름은 프로그램 전체에서 사용할 수 있지만, *한 가지 중요한 제약이 있다!*

> ❗ **순환 참조는 엄격히 금지된다!** 타입과 값 정의 모두 마찬가지다.
>
> 즉, `Alice`라는 타입에서 `Bob` 타입을 사용했다면 `Bob`에서는 `Alice`를 사용할 수 없다. 함수나 다른 정의의 경우에도 마찬가지다. 사실 `Alice` 자신도 `Alice`를 사용할 수 없다!

얼핏 터무니없어 보이는 제약이지만, 중요한 동기와 혁신적인 해결책이 있다.

이 제약의 **동기**는 바로 **전체성**(totality)이라는 Par의 야심찬 목표이다. 즉, 무한 루프를 방지한다는 것이다.

무제한적인 재귀는 무한루프의 원인이 되며 [Agda](https://ko.wikipedia.org/wiki/Agda)에서와 같이 전체성 확인을 통해 부분적으로 해소할 수 있지만, Par에서는 다른 접근을 취한다. 무제한 재귀를 금지하고, 원래 상호 재귀를 사용해 구현하는 동작을 포함해 모든 순환 동작을 더욱 *원칙적인* 방법으로 구현하는 것이다.

바로 이 원칙적인 방법이 이 제약의 **해결책**이 된다. 물론 이유 없이 재귀가 복잡해지기만 하는 것은 아니고, 이 방법 역시 나름대로의 장점이 있다.

값 수준(즉, 함수 등에서)에서는 초보적 재귀 대신 **`begin`/`loop`라는 강력한 보편적 반복 메커니즘**이 존재하며, 다음 상황에 모두 사용할 수 있다.
- *[재귀](../types/recursive.md)적 축약*으로써 리스트나 트리뿐만 아니라 파일도 분석할 수 있다.
- *[반복](../types/iterative.md)적 생성*으로써 연거푸 상호작용할 수 있는 개체를 만들 수 있다.
- [프로세스 문법](../process_syntax.md)에서는 *명령헝처럼 보이는 루프*도 작성할 수 있다.

타입의 초보적 재귀 대신에는 **익명 [재귀](../types/recursive.md)와 [반복](../types/iterative.md) (쌍대재귀) 타입**이 존재한다.

## 정의

(함수를 포함해서) 전역 값은 파일의 최상위에 키워드 `def`, 대문자 이름, 등호, 원하는 값을 계산하는 식을 순서대로 작성해 정의할 수 있다.

```par
module Main

def MyNumber = 7
```

이때 Par는 `MyNumber`의 타입을 [`Nat`](./primitive_types.md)(자연수)로 추론할 수 있으므로 타입 표기는 하지 않아도 되지만, 타입 표기를 명시적으로 작성할 필요가 있을 때는 이름 뒤에 쌍점을 붙이고 작성할 수 있다.

```par
def MyName: String = "Michal"
```

## 선언

타입이 지나치게 길어져 정의를 읽기 어려워질 때도 종종 있을 수 있다.

예를 들어, 리스트의 모든 수를 합하는 간단한 함수를 살펴보자.

```par
def SumList: [List<Int>] Int = [list] list.begin.case {
  .end!       => 0,
  .item(x) xs => x + xs.loop,
}
```

이 코드에서는 아직 다루지 않은 여러 가지 개념을 사용하므로, 이미 배운 `def`, `:`, `=` 부분에만 집중해 보자. _(물론 실제로 플레이그라운드에서 실행하는 것은 가능하다!)_

이 경우에는 타입 표기를 별도의 *선언*으로 추출할 수 있다. 선언은 키워드 `dec`, 표기의 대상이 되는 이름, 쌍점, 타입을 순서대로 작성하면 된다.

```par
dec SumList : [List<Int>] Int

def SumList = [list] list.begin.case {
  .end!       => 0,
  .item(x) xs => x + xs.loop,
}
```

한결 나아졌다!

선언은 파일의 아무 곳에 작성해도 무관하므로, 파일의 최상단으로 몰아서 작성하거나 정의와 가까운 곳에 붙여도 된다.

## 타입 정의

Par의 타입 시스템은 [구조적이다](../introduction.md#얕지만-넓은-직교성). 많은 언어에서 여러 형태의 타입 정의를 지원하지만(예를 들어, Rust에는 `struct`나 `enum` 따위가 있다), Par가 지원하는 것은 **타입 동의어** 하나뿐이다.

> [재귀](../types/recursive.md)/[반복](../types/iterative.md) 타입은 익명이므로 Par에서는 타입을 이름 대신 형태로 취급하는 데 문제가 없다. 더 나아가, Par에서는 모든 타입 정의가 불필요하다. [원시 타입](./primitive_types.md)을 제외하면 타입 동의어가 더 이상 남지 않을 때까지 정의로 계속 치환해도 전혀 문제가 없다.
>
> *사실 Par에서는 정의라는 개념 자체가 불필요하다. 물론 정의 없이 프로그래밍을 한다면 매우 귀찮을 것이다.*

타입에 이름을 붙이려면 파일의 최상위에서 키워드 `type`, 대문자 이름, 등호 다음에 그 이름을 붙일 타입을 작성하면 된다.

```par
type MyString = String
```

이제 `MyString`은 `String`과 같으며, `String`이 올 수 있는 곳에 어디든 사용할 수 있다.

더 유용한 예시는 다음과 같다.

```par
type StringBuilder = iterative choice {
  .build => String,
  .add(String) => self,
}
```

> 해당 타입은 `String.Builder`라는 내장 기능에서 사용하고 있다.

이 타입은 나중에 다룰 [반복](../types/iterative.md) [선택](../types/choice.md) 타입으로, 연거푸 상호작용할 수 있으며 사용할 때마다 분지(메서드)를 선택할 수 있다.

물론 `StringBuilder`를 사용할 때마다 매번 위의 정의를 복사할 수 있겠지만, 굳이 이렇게 할 이유는 없을 것이다.

### 제네릭 타입

타입 정의에 제네릭 타입 매개변수를 추가로 작성하면 매개변수로 전달한 타입으로 인스턴스화할 수 있는 공식으로 만들 수 있다.

타입 매개변수는 타입 이름 바로 뒤에 반점으로 구분하고 부등호로 감싸서 작성할 수 있다. 매개변수는 지역 이름이므로 소문자로 작성해야 한다.

예를 들어, 내장된 `List` 타입에는 하나의 매개변수가 있다.

```par
type List<a> = recursive either {
  .end!,
  .item(a) self,
}
```

이 타입은 역시 나중에 다룰 [재귀](../types/recursive.md) [분기](../types/either.md) 타입으로, `a` 타입의 원소를 가지는 유한한 단일 연결 리스트를 정의한다.

제네릭 타입을 사용할 때도 타입 이름 뒤에 타입 인자를 반점으로 구분하고 부등호로 감싸서 작성하면 된다.

```par
type IntList = List<Int>
```

제네릭 타입을 사용하면 타입 정의에서 각각의 타입 변수를 대응하는 타입 인자로 바꾼 타입이 된다. 즉, 위의 타입은 아래와 같다.

```par
type IntListExpanded = recursive either {
  .end!,
  .item(Int) self,
}
```

이제 모듈이 가질 수 있는 내용물을 알아봤으니, 조금 더 먼발치에서 **패키지, 모듈, 가져오기, 내보내기**에 대해 알아보자.
